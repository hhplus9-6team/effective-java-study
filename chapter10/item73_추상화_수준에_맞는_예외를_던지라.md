# item73. 추상화 수준에 맞는 예외를 던지라

### 왜 추상화 수준에 맞는 예외를 던져야 하는가?
하위 계층(저수준)에서 발생한 예외를 그대로 상위 API로 전파하면 다음 문제가 발생한다.
- 추상화 누수 : 상위 계층의 책임과 무관한 예외 타입이 외부로 노출된다.
- API 오염 : 호출자는 내부 구현을 알아야 예외를 이해할 수 있다.
- 유지보수성 저하 : 내부 구현 변경 시 외부 API 의 예외 계약까지 흔들린다.


### 예외 번역
- 상위 계층에서 저수준 예외를 잡아 의미 있는 고수준 예외로 변환한다.
- 저수준 예외는 상위 API의 관점에서 재해석되어야 한다.
- 예: SQLException → OrderPersistenceException
```
try {
    ... //저수준 추상화를 이용한다.
} catch (LowerLevelException e) {
    //추상화 수준에 맞게 번역한다.
    throw new HigherLevelException(...);
}
```

### 예외 연쇄
- 저수준 예외가 디버깅 가치가 있다면 버리지 말고 원인(cause) 으로 포함한다.
- Throwable.getCause()로 근본 원인 접근 가능
  - 고수준 예외의 생성자는 예외 연쇄용으로 설계된 상위 클래스의 생성자에 원인을 건내주어 최종적으로 Throwable 생성자까지 건네지게 된다.
- 스택 트레이스가 자연스럽게 통합되어 장애 분석에 유리

```
try {
  ...//저수준 추상화를 이용한다.
} catch (LowerLevelException cause) {
  //저수준 예외를 고수준 예외에 실어 보낸다.
  throw new HigherLevelException(cause);
}
```
- 대부분의 표준 예외는 예외 연쇄용 생성자를 갖추고 있다. 
- 그렇지 않은 예외도 Throwable initCause 메서드를 이용해 원인을 직접 못박을 수 있다.
  
### 예외 번역보다 더 좋은 선택지
- 저수준 메서드가 반드시 성공하도록 하여 아래 계층에서는 예외가 발생하지 않도록 하는 것이 최선이다.
- 상위 계층 메서드의 매개변수 값을 아래 계층 메서드로 건네기 전에 미리 검사하는 방법으로 달성할 수 있다.
- 상위 계층에서 그 예외를 조용히 처리하여 문제를 API 호출자에까지 전파하지 않는 방법이있다.
